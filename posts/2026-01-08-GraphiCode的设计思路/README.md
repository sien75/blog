<h1>GraphiCode 的设计思路</h1>

## 目前的 AI Coding 工具无法胜任大型项目

目前的 Lovable / Replit / Cursor 等工具, 已经验证了**大模型具备局部代码的生成能力**, 不过这还不足以胜任复杂的大型项目.

对于真实场景下复杂的大型项目, 由于代码的复杂性和项目的真实性, **很难完全相信大模型产出的代码结果**, 这时其实就面临一个很头疼的问题: 大模型改代码是挺快, 但是验证这些改动对不对 / 会有什么影响却非常困难.

换句话说, 在大模型的加持下, **代码的 generation 成本低, 但是 review 成本高**.

那为什么无法完全信任大模型产出, 反而还会陷入具体的代码细节呢? 是大模型的能力不足, 还是说目前还没有找到一种方式, 可以让人避免关注到大部分代码细节, 只关注到宏观的架构设计?

## 分析当前的开发模式

观察一下当前的开发模式, 这其中有几个值得思考的点.

### 1st

第 1 点, 编程语言代码是一种文本, 文本的表达成本低, 但是理解成本高, 结构化能力不足. 相对应的, 图的表达成本高, 远不如打几个字方便, 但是人一定更喜欢看图而不是文字, 因为图更直观.

简而言之, **人总是喜欢"看图, 说话"**.

所以这里有一个问题: **文本的理解成本高, 是不是代码 review 成本高的一个重要因素呢?**

### 2nd

第 2 点, 当前的项目开发中**普遍存在调试/测试成本高的情况**. 这并不是耸人听闻, 大家熟悉的方式未必是最合理的.

实际上, 在当前的项目开发模式中, 如果想针对某一个模块的代码进行测试 -- 无论是验证代码正确性还是复现问题, 那都得花心思写代码构造合适的 Mock 数据, 更不用说线上的问题排查场景, 需要专门花心思埋日志, 然后从庞杂的日志中找到问题的原因.

### 3rd

第 3 点, 在当前的人员分工下, 存在两种**繁杂且效率低下的沟通**, 即 UI 验收和技术联调, 这个想必开发同学都清楚这一点. 事实上, 当前都是不同职能的人通过文档 / 设计稿 / API 工具这些外部系统去进行对齐的, 存在一个 A 的内部 -> 外部 -> B 的内部的过程, 这导致有很多时间浪费在了不必要的沟通上.

在当前大模型的加持下, 这其实意味着**职能划分需要做一些调整**, 确保相似职能内聚到同一个职能上, 然后不同职能之间有便利的对接方式.

> 这里讨论的职能划分不涉及真实人员组织上的变动, 而是要设计更加合理的工具去确保职能划分的合理

## New Pattern: Flowchart + Natural Language + LLM

这里介绍一种新的开发模式: **借助于 LLM 能力, 用流程图 + 自然语言进行编程**.

为了实现这种开发模式, 个人计划开发一款工具, 就叫做 <span style="color: red">**GraphiCode**</span>.

### Flowchart

一个软件项目中, 本来就会有复杂的逻辑关系/依赖关系, 这些关系更适合用流程图来表示. **流程图的好处, 主要是方便 review, 无论是自己还是其他人都能快速理清逻辑.**

另外, 虽然整体上来说, 生成图比生成文字的成本要高, 但是流程图并非真正的图, 它可以通过 DSL (如 Mermaid) 或者专门的流程图工具 (如 Drawio) 去快速生成. 事实上, 很多人在编码之前都有先画流程图分析的习惯.

对于流程图来说, 真正需要关注的问题, 是流程图一旦复杂起来, 就会显得非常乱. 这也是完全用流程图表达逻辑的系统都会遇到的问题. 要避免这个问题, **就需要将流程图的能力做窄, 让流程图作为统筹整体的工具, 而不是随便一个 if 关系都要画到流程图里.**

> 因此和 Node-RED 这类的低代码平台不同, GraphiCode 并不致力于不写代码, 也不会将逻辑全部扔到流程图里, 也不会提供一堆输入框去做"配置化", 而是找到一种平衡方式, 有效结合流程图和代码.

### Programming = Data + Algorithm

下面说一下, 应该如何做到这种平衡方式.

大家都知道, 从技术上讲, 编程 = 数据 + 算法. 数据代码和算法代码的特点是非常不一样的, 具体来说, **数据代码偏向于声明式和映射思维, 算法代码偏向于命令式和逻辑思维.**

比如:

* 在编写 UI 界面代码时, 主要想的是这个地方应该放置什么元素, 怎么去对应设计稿的图形, 这是声明式和映射思维.
* 在编写数据格式转换代码时, 主要想的是应该如何处理输入数据, 从哪里取值, 做什么逻辑转换, 然后组装成什么输出格式, 这是命令式和逻辑思维.

当然开发时的思维方式不是最关键的, 最关键的是这两类代码的 review 成本是不同的: **数据代码是好 review 的, 算法代码是不好 review 的.**

因此一个很自然的思路是, **把数据代码和算法代码完全拆开**, 数据格式定义 / 数据状态实体都属于数据代码, 算法函数属于算法代码. (这有点类似于数据驱动编程)

针对算法代码不好 review 的问题, 试想一下, 算法代码真的能通过 review 判断是不是写对了吗? 其实, **算法代码的编写是否正确, 一定是测出来的, 而不是看出来的.** 因此, 在大模型生成代码的当下, 一个纯算法代码是否正确, 只需要进行功能测试和性能测试即可, 并不需要去 review 里面的逻辑.

### Rethinking Flowchart

这里探讨深入一下, 思考如何设计流程图.

首先, **流程图需要对应到一种 DSL (例如 Mermaid), 最终以代码的方式管理.** 针对这个 DSL, 一方面需要能进行渲染方便人去看, 另一方面需要设计方法去解析这个 DSL, 获取到 DSL 表示的模块关系.

流程图内的模块分类如下:

* 数据模块
  * 数据格式定义
  * 数据状态实体
* 算法模块
  * 算法函数

`数据格式定义`, 指`interface` / `type` / `class` 一类数据, 没有真正的 instance, 只是声明了数据的格式. `数据状态实体`, 指`memory_data` / `storage_data` / `database_data`, 表示真正占用了内存/硬盘/数据库的数据.

`算法函数`, 应该是只依赖语言引擎的纯函数, 并且包含一些对外部的接口, 比如输入数据 / 输出作用, 这些都会链接到对应的数据状态实体.

**将算法函数按照需求串接起来, 就能够形成流程图, 而数据模块"堆到一边", 作为一个参考资料即可, 完全不必参与流程图的串接.** 这是 GraphiCode 流程图的最基本的雏形.

### Automate The Algorythm Module

费劲设计这么一套流程图方案, 除了方便数据模块的 Review, 最重要的, 是**方便了算法模块的调试/测试.**

因为调试/测试其实是针对算法模块的工作, 这套流程图方案里的算法模块都是纯函数, 而纯函数是非常便于反复测试的.

知道大家都不喜欢写测试代码, 但是别忘了, 还有大模型. 所以设想一下算法模块的功能:

1. 大模型首先理解你的需求, 把你的口语转换成一份自然语言的逻辑描述, 描述这个函数要干啥, 输入是什么数据格式, 输出是什么数据格式
2. 大模型还会根据自然语言描述生成一份编程语言的函数实现, 当然得是纯函数
3. 大模型还会依据自然语言自动生成一份详细的测试用例和 Mock 数据, 包括各种边界情况, 去测试这个编程语言实现的函数

**只要自动测试通过, 那么这个算法模块就完全不用担心正确性.** 算法模块自身可以做成 Agent 式的"测试结果->更改代码"的反馈模式, 即: 改到测试用例都通过为止.

而这一切, 是自动完成的.

那么人的心智负担就从 review 代码变为了"理解这个算法模块干了啥", 那么其实你只需要去阅读这个算法模块的自然语言描述, 去使用现成的 Mock 数据试一试, 就知道这个算法模块的作用了.

### Generalize The Data Module

聊完了算法模块, 回过头来看一下数据模块. 如果程序只是一堆算法, 那就是空中楼阁, 只有作用到具体的事物上才有用.

在这里, 需要**泛化一下"数据模块"的概念.** 之前提到过, 数据模块的两个重要概念是"声明"和"映射". 比如说, React 函数式组件, 本身声明了 JSX, 这个 JSX 其实是能映射到浏览器 DOM 结构, 最终映射到显示设备.

因此, React 函数式组件就可以认为是一个数据模块. 同样的, 数据库也可以认为是一个数据模块.

那么, 考虑到刚才提到的 UI 验收困难的问题, 更好的方式是, 设计师设计完成后直接输出组件(这件事有很多人在做), 那么这个组件就是 GraphiCode 里的数据模块.

再这样的架构下, 微调 UI 完全是设计师的职责, 工程师需要关注的是, 组织好合适的数据格式, 把数据填充给 UI 模块即可.

数据模块一样可以使用上述算法模块"测试"的概念, 比如: 设计师觉得样式不合理, 那他可以自己调整好样式, 使用生成好的 mock 数据去测试各种场景, 只要各种场景的显示符合预期, 那这个 UI 数据模块就是没问题的, 直接发布即可.

### Multi Language Support

在实际的生产环境中, 往往存在多种语言环境, 比如后端使用 Java, 前端使用 JavaScript, 客户端使用 Swift / Kotlin.

如果想要提升沟通效率, 比如前面提到的联调效率, 那一定要做的一件事就是**统一数据定义方式**, 防止不断的重复定义.

因此 GraphiCode 计划**使用 Protobuf 作为数据模块的默认数据定义方式.**

一旦采用了 Protobuf, 那么 GraphiCode 其实就能天然支持多语言. 每种语言都有自己擅长的场景, 在同一个项目中使用自然语言编程, 而实际上底层对应了不同的编程语言, 这想想都是一件很神奇的事情.

因此可以说, **自然语言是一种编程语言, LLM 就是自然语言的编译器, 以往都是把提示词扔进去, 拿最终的代码结果, 但是自然语言文本本身, 不就是一种代码吗?**

### About Reuse

复杂项目必须考虑复用(或者叫组件化)的事情, 刚才提到的数据模块和算法流程, 应该如何组装复用呢?

这里不妨思考一下, 数据模块和算法模块各自有什么原则? 数据模块可以有副作用, 对外有接口(读/改/订阅), 而算法模块是纯函数, 最好不要有副作用.

所以如果某个更大维度的模块 A, 包括数据模块 + 算法流程, 那么如果算法模块的输入和输出只会作用到 A 模块下的数据模块, 那么其实这只是一种内部作用. 因此, 外部并不需要关注到 A 模块内部的算法部分, 而只需要关注到 A 模块的数据模块接口. 因此, 从宏观上来看, A 模块是数据模块.

也就是说, **数据模块 + (纯内部作用的)算法流程 = (新的)数据模块.**

另外, 算法流程当然也可以抽象为单一算法模块, 这其实就是函数式编程中的 Compose 的概念.

也就是说, **算法流程 = (一个复杂的)算法模块.**

有了这一套概念, 其实就做到了多层套用, 这样就能支持复杂的项目了.
